# Installation Guide

This guide covers everything you need to install, configure, and run the Sinapsis Marketing Campaigns API in different environments.

## Prerequisites

Before you begin, ensure you have the following installed on your system:

### Required Software

Check that all prerequisites are installed correctly:

```bash
# Check Docker version
docker --version
docker-compose --version

# Check Bun version
bun --version
```

  While the project uses Bun for development, you can also use npm or pnpm. However, Bun provides the fastest install and runtime performance.
</Tip>

## Installation Steps

<Steps>
  <Step title="Clone the Repository">
    Download the source code from GitHub:

    ```bash
    git clone https://github.com/JonathanLara27/backend-sinapsis-test
    cd backend-sinapsis-test
    ```
  </Step>

  <Step title="Install Dependencies">
    Install all required packages:

    <CodeGroup>
    ```bash Bun (Recommended)
    bun install
    ```

    ```bash npm
    npm install
    ```

    ```bash pnpm
    pnpm install
    ```
    </CodeGroup>

    This installs:
    - **NestJS** - Core framework and modules
    - **TypeORM** - Database ORM with MySQL driver
    - **Serverless** - Framework and offline plugin
    - **Validation** - class-validator and class-transformer
    - **Testing** - Jest and related testing utilities
  </Step>

  <Step title="Configure Environment Variables">
    The application uses environment variables for configuration. For local development, you can rely on the default values, or create a `.env` file:

    ```bash .env
    # Database Configuration
    DB_HOST=localhost
    DB_PORT=3307
    DB_USERNAME=root
    DB_PASSWORD=root
    DB_NAME=sinapsis_db
    DB_SYNCHRONIZE=true

    # Swagger Documentation
    SWAGGER_TITLE=Sinapsis API
    SWAGGER_DESCRIPTION=Marketing Campaigns Management API
    SWAGGER_VERSION=1.1.1
    SWAGGER_PATH=api/docs

    # Deployment Stage
    STAGE=dev
    ```

    <Warning>
      **Never set `DB_SYNCHRONIZE=true` in production!** This automatically synchronizes your schema and can lead to data loss. Only use this in development.
    </Warning>
  </Step>

  <Step title="Start the Database">
    Launch MySQL using the provided Docker Compose configuration:

    ```bash
    docker-compose up -d
    ```

    The `docker-compose.yml` configuration:

    ```yaml docker-compose.yml
    services:
      db:
        image: mysql:8.0
        restart: always
        environment:
          MYSQL_ROOT_PASSWORD: root
          MYSQL_DATABASE: sinapsis_db
        ports:
          - '3307:3306'
        volumes:
          - db_data:/var/lib/mysql

    volumes:
      db_data:
    ```

    <Info>
      The database runs on **port 3307** (mapped from container port 3306) to avoid conflicts with existing MySQL installations on your machine.
    </Info>

    Verify the container is running:

    ```bash
    docker ps
    ```

    You should see output similar to:

    ```
    CONTAINER ID   IMAGE       PORTS                    STATUS
    abc123def456   mysql:8.0   0.0.0.0:3307->3306/tcp   Up 10 seconds
    ```
  </Step>
</Steps>

## Running the Application

The API supports multiple runtime modes depending on your use case.

### Development Mode (Recommended for Local Development)

Run the NestJS application directly with hot reload:

```bash
bun run start:dev
```

This mode:
- Uses Bun's native `--watch` flag for instant reloads
- Runs `src/main.ts` directly without building
- Exposes API at `http://localhost:3000/api`
- Serves Swagger docs at `http://localhost:3000/api/docs`

<Tip>
  Development mode is the fastest way to iterate on code changes with automatic reloading.
</Tip>

### Serverless Offline Mode (Lambda Simulation)

Simulate AWS Lambda environment locally:

```bash
bun run start:sls
```

This mode:
1. Builds the TypeScript code with `nest build`
2. Starts `serverless-offline` to simulate API Gateway and Lambda
3. Exposes API at `http://localhost:3000/dev`
4. Serves Swagger docs at `http://localhost:3000/dev/api/docs`

<Note>
  The `/dev` prefix comes from the serverless stage configuration. This matches how the API will behave when deployed to AWS.
</Note>

### Production Mode (Built Application)

Run the compiled JavaScript code:

```bash
# Build the application
bun run build

# Run the production build
bun run start:prod
```

### Available Scripts

Here are all the npm scripts defined in `package.json`:

| Script | Command | Description |
|--------|---------|-------------|
| `start:dev` | `bun --watch src/main.ts` | Development mode with hot reload |
| `start:sls` | `bun run build && bun x serverless offline` | Serverless offline mode |
| `start:prod` | `bun dist/main` | Run production build |
| `build` | `nest build` | Compile TypeScript to JavaScript |
| `test` | `bun run jest` | Run unit tests |
| `test:cov` | `jest --coverage` | Run tests with coverage report |
| `lint` | `eslint` | Lint and fix code |
| `format` | `prettier --write` | Format code |

## Database Configuration

### TypeORM Connection

The application uses TypeORM to connect to MySQL. Configuration is loaded from environment variables:

```typescript
{
  type: 'mysql',
  host: process.env.DB_HOST,      // localhost
  port: process.env.DB_PORT,      // 3307
  username: process.env.DB_USERNAME, // root
  password: process.env.DB_PASSWORD, // root
  database: process.env.DB_NAME,  // sinapsis_db
  synchronize: process.env.DB_SYNCHRONIZE, // true in dev only
  entities: ['dist/**/*.entity.js'],
}
```

### Schema Synchronization

When `DB_SYNCHRONIZE=true`, TypeORM automatically:
- Creates tables from entity definitions
- Updates schema when entities change
- Adds or removes columns as needed

<Warning>
  **Production Warning**: Always set `DB_SYNCHRONIZE=false` in production and use migrations instead. Auto-sync can cause data loss.
</Warning>

## Serverless Configuration

The `serverless.yml` file configures deployment to AWS Lambda:

```yaml serverless.yml
service: sinapsis-backend
frameworkVersion: '3'

provider:
  name: aws
  runtime: nodejs18.x
  region: us-east-1
  stage: ${opt:stage, 'dev'}
  timeout: 30
  environment:
    # Environment variables loaded from .env
    DB_HOST: ${env:DB_HOST}
    DB_PORT: ${env:DB_PORT}
    DB_USERNAME: ${env:DB_USERNAME}
    DB_PASSWORD: ${env:DB_PASSWORD}
    DB_NAME: ${env:DB_NAME}
    DB_SYNCHRONIZE: ${env:DB_SYNCHRONIZE}
    SWAGGER_TITLE: ${env:SWAGGER_TITLE}
    SWAGGER_DESCRIPTION: ${env:SWAGGER_DESCRIPTION}
    SWAGGER_VERSION: ${env:SWAGGER_VERSION}
    SWAGGER_PATH: ${env:SWAGGER_PATH}

functions:
  main:
    handler: dist/main.handler
    events:
      - http:
          method: ANY
          path: /
      - http:
          method: ANY
          path: '{proxy+}'
```

### Serverless Offline Plugin

The `serverless-offline` plugin simulates API Gateway and Lambda locally:

```yaml
plugins:
  - serverless-offline
```

When running `bun run start:sls`, this plugin:
- Starts a local HTTP server
- Routes requests through the Lambda handler
- Simulates cold starts and warm invocations
- Provides logs similar to CloudWatch

## Troubleshooting

<AccordionGroup>
  <Accordion title="Port 3307 already in use">
    If you have another MySQL instance on 3307, edit `docker-compose.yml` to use a different port:

    ```yaml
    ports:
      - '3308:3306'  # Changed from 3307
    ```

    Then update `DB_PORT=3308` in your `.env` file.
  </Accordion>

  <Accordion title="Docker container won't start">
    Remove existing containers and volumes:

    ```bash
    docker-compose down -v
    docker-compose up -d
    ```
  </Accordion>

  <Accordion title="TypeORM connection errors">
    Verify your environment variables match the Docker configuration:
    
    - `DB_HOST=localhost` (not 127.0.0.1)
    - `DB_PORT=3307`
    - `DB_USERNAME=root`
    - `DB_PASSWORD=root`
    - `DB_NAME=sinapsis_db`
  </Accordion>

  <Accordion title="Serverless offline mode fails">
    Make sure you build the project first:

    ```bash
    bun run build
    ```

    Then try running serverless offline again:

    ```bash
    bun x serverless offline
    ```
  </Accordion>
</AccordionGroup>
